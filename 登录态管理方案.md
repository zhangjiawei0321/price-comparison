# 登录态管理统一方案（price-comparison）

## 目标

- 让监控任务在长期运行时优先复用已登录态，减少因未登录导致的首页跳转、验证码页和空价格。
- 将“登录态问题”从“抓取失败”里拆分出来，形成可观测、可恢复、可操作的闭环。
- 保持现有架构不大改：基于 Playwright `userDataDir` + 现有风控判定与告警能力。

---

## 当前实现现状（已具备能力）

你当前仓库已经有一套可用基础能力：

- `index.js` 统一解析 `--userDataDir` / `PRICE_USER_DATA_DIR`。
- `src/scraper.js` 使用 `chromium.launchPersistentContext(userDataDir, ...)` 复用登录态。
- `src/scraper.js` 有 `htmlWasCaptcha` 判定（落到首页/登录页/验证页会识别）。
- `src/monitor.js` 在 `add` 与 `monitor` 两条路径都对风控状态做了拦截和告警。
- `src/scraper.js` 有同一 `userDataDir` 的串行锁（避免 profile 被并发占用）。

结论：你现在不是“没有登录态方案”，而是“有基础方案，缺统一使用规范和运维流程”。

---

## 统一方案（建议落地版）

### 1) 登录态存储层：单独 profile + 目录持久化

- 本机运行：固定一个目录给 `PRICE_USER_DATA_DIR`，不要用日常 Chrome 的 Default 目录。
- Docker 运行：固定映射到宿主机持久目录（你现在是 `docker-data/profile`，保持即可）。
- 不要在多个进程同时占用同一个 `userDataDir`（已通过代码锁做进程内串行，但跨进程仍需人为避免）。

推荐：

```env
PRICE_USER_DATA_DIR=C:\Users\<你>\price-monitor-profile
```

### 2) 登录态获取层：一次人工 headful 登录，长期复用

- 首次初始化或失效后，使用 `--headful` 打开页面，人工完成登录/验证。
- 成功后关闭该次任务；后续监控默认无头（`headful=false`）复用同一 profile。
- 仅在告警提示“风控/登录态失效”时再临时 headful 续期。

### 3) 登录态健康判定：抓取前不额外探针，抓取后统一判定

沿用现有策略，不新增独立“登录检测 API”：

- 真实抓取后，基于 `finalUrl + html + hasReliablePrice` 得到 `htmlWasCaptcha`。
- 触发则本轮不入库，并发送风险告警（当前已实现）。

优点：简单、跨站点统一、和业务成功标准一致（“能抓到可靠价”）。

### 4) 失败处理策略：分级回退

建议按以下顺序处理：

1. 单次失败：自动重试（你已有 `SCRAPE_RETRIES` 逻辑）。  
2. 同轮连续失败：跳过写库，仅发风控提醒（你已实现）。  
3. 多轮持续失败：人工 `--headful` 重新登录并确认页面可见真实价格。  

可操作阈值建议：

- `RISK_WEBHOOK_COOLDOWN_SECONDS=3600`（默认即可，避免告警轰炸）
- `JD_MONITOR_GAP_SECONDS=30`（当前默认值合理，降低连续访问压力）

### 5) 并发与调度策略：单进程主循环优先

- 优先使用一个进程承载 `web --monitor` 或 `run`，避免多个监控实例抢同一 profile。
- 若必须多实例，给每个实例分配不同 `PRICE_USER_DATA_DIR`（相互隔离）。

### 6) 运维 SOP（建议固定流程）

- **初始化**
  - 配置 `PRICE_USER_DATA_DIR`
  - `node index.js add "<商品URL>" --headful`
  - 确认可以成功解析价格后再开定时监控

- **日常运行**
  - 常态 `headful=false`
  - 观察是否出现 `[warn] maybe blocked/captcha`

- **告警后恢复**
  - 暂停监控
  - `node index.js debug-jd "<URL>" --headful` 或 `debug-tb ... --headful`
  - 页面确认正常后恢复监控

## 与参考仓库的对齐点

参考的 3 个项目本质都是“会话持久化 + 有效性校验 + 失效后人工介入”：

- `jd_seckill`：手工 cookies 注入 + 校验是否有效。
- `jd-assistant`：本地 cookies 持久化（pickle）+ 启动校验 + 失效扫码。
- `JdBuyer`：同样是本地 cookies 持久化 + 校验 + 扫码续期。

你的项目选择的是 Playwright 持久 profile（不是 pickle cookie 文件），思路一致，且更适合多站点页面渲染场景。

## 最小改造清单（可选）

下面 3 项可作为下一步增强（非必须）：

1. 新增 `npm run login:init`（封装一次 headful 初始化流程）。
2. 在 Web UI 增加“登录态自检”按钮（调用一次 `parseProductOnce` 并显示 `htmlWasCaptcha`）。
3. 增加“连续 N 轮风控自动暂停”开关，避免长时间无效重试。

## 验收标准

满足以下条件即可认为方案落地成功：

- 同一商品连续监控 24h 内，大多数轮次能稳定写入有效价格点。
- 出现登录/验证页时，不写入脏价格，并能收到风险提醒。
- 人工一次 `--headful` 处理后，监控可恢复到无头稳定运行。
